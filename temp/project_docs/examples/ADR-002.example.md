# ADR-002: Temporal Normalization для PersonRelationship

> **Статус:** Accepted
> **Дата:** 2026-04-17
> **Автор:** Дмитрий Бондарев
> **Теги:** `database` `domain-model` `temporal`

---

## Контекст

Модель PersonRelationship хранит связи между персонами (parent→child, spouse, sibling). В исходной схеме связи не имели временного измерения — невозможно было отразить, что связь действовала в определённый период (например, брак с 1975 по 1990). Для корректного отображения семейного дерева на временной шкале и для работы с Unions необходимо добавить temporal-поля.

### Движущие силы (Drivers)

- **Историческая точность:** Семейные связи меняются во времени (браки, разводы, усыновления)
- **Union ↔ Relationship синхронизация:** start_date/end_date Union должны отражаться в valid_from/valid_to связей spouse-типа
- **Timeline UX:** Для визуализации «кто кому кем был когда» нужны временные границы

---

## Рассмотренные варианты

### Вариант A: Добавить valid_from / valid_to в PersonRelationship

| Критерий          | Оценка     |
|-------------------|------------|
| Сложность         | Low        |
| Производительность| Минимальное влияние (2 nullable DATE-поля) |
| Поддержка         | Стандартный паттерн temporal modeling       |

**Плюсы:**
- Минимальные изменения схемы
- Обратная совместимость (NULL = действует всегда)
- Стандартный подход в industry

**Минусы:**
- Требует backfill существующих данных

### Вариант B: Отдельная таблица relationship_periods

| Критерий          | Оценка     |
|-------------------|------------|
| Сложность         | High       |
| Производительность| JOIN при каждом запросе |
| Поддержка         | Сложнее для понимания   |

**Плюсы:**
- Можно хранить несколько периодов для одной связи

**Минусы:**
- Избыточная нормализация для текущего масштаба
- Сложность запросов

---

## Решение

**Выбран вариант A: поля valid_from / valid_to в PersonRelationship.**

Простой и расширяемый подход. NULL в обоих полях означает «связь действует всегда» — обратная совместимость сохранена. Backfill данных выполняется в два этапа: сначала death_date → valid_to, затем Union dates → spouse valid_from/valid_to.

---

## Последствия (Consequences)

### Положительные

- Можно точно отобразить «брак 1975–1990» на Timeline
- Запросы вида «кто был в браке на дату X» становятся тривиальными
- Подготовка к визуальному дереву с временной шкалой

### Отрицательные

- Необходимость аккуратного backfill (ручная проверка данных)
- NULL-семантика требует документирования (NULL = бессрочно)
