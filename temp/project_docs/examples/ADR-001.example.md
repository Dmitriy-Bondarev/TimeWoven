# ADR-001: Миграция с SQLite на PostgreSQL

> **Статус:** Accepted
> **Дата:** 2026-04-16
> **Автор:** Дмитрий Бондарев
> **Теги:** `database` `infrastructure` `migration`

---

## Контекст

TimeWoven использовал SQLite как основную БД с момента создания. По мере роста проекта стали проявляться ограничения: невозможность одновременного доступа (concurrent writes), отсутствие встроенных типов данных (JSONB, ARRAY, INTERVAL), ограниченные возможности миграций и масштабирования. Запуск продакшен-сервера на VPS (Hostkey) с перспективой многопользовательского доступа сделал миграцию необходимой.

### Движущие силы (Drivers)

- **Concurrency:** SQLite блокирует файл при записи — невозможно обслуживать несколько пользователей одновременно
- **Масштабируемость:** Планируемые фичи (Telegram Bot, визуальное дерево, collaborative editing) требуют полноценную СУБД
- **Типизация данных:** Temporal-поля (valid_from, valid_to), JSONB для метаданных воспоминаний
- **Продакшен-готовность:** Бэкапы (pg_dump), репликация, мониторинг — всё out of the box

---

## Рассмотренные варианты

### Вариант A: Остаться на SQLite

| Критерий          | Оценка     |
|-------------------|------------|
| Сложность         | Low        |
| Производительность| Ограничена при concurrent writes |
| Поддержка         | Минимальная инфраструктура       |

**Плюсы:**
- Нулевые затраты на миграцию
- Простота локальной разработки (один файл)

**Минусы:**
- Потолок масштабирования уже достигнут
- Нет concurrent writes
- Ограниченные типы данных

### Вариант B: PostgreSQL 14+

| Критерий          | Оценка     |
|-------------------|------------|
| Сложность         | Medium     |
| Производительность| Отличная для текущих и будущих нагрузок |
| Поддержка         | Зрелая экосистема, документация, инструменты |

**Плюсы:**
- Полная поддержка concurrent access
- Богатые типы данных (JSONB, temporal, arrays)
- pg_dump для бэкапов, готовность к репликации
- SQLAlchemy 2.0 работает прозрачно с обоими движками

**Минусы:**
- Необходимость миграции данных
- Дополнительный сервис на сервере (PostgreSQL daemon)
- Настройка pg_hba.conf, пользователей, прав

### Вариант C: MySQL / MariaDB

| Критерий          | Оценка     |
|-------------------|------------|
| Сложность         | Medium     |
| Производительность| Хорошая    |
| Поддержка         | Широкая, но JSONB менее зрелый чем в PG |

**Плюсы:**
- Популярна, много хостингов поддерживают «из коробки»

**Минусы:**
- Менее богатая типизация по сравнению с PostgreSQL
- JSONB-поддержка уступает PostgreSQL
- Нет преимуществ перед PostgreSQL для нашего use case

---

## Решение

**Выбран вариант B: PostgreSQL 14+.**

PostgreSQL полностью закрывает текущие и прогнозируемые потребности TimeWoven. SQLAlchemy 2.0 обеспечивает прозрачный переход — модели остаются без изменений, меняется только `DATABASE_URL`. Богатые типы данных (temporal, JSONB) критичны для планируемой temporal normalization и метаданных воспоминаний.

---

## Последствия (Consequences)

### Положительные

- Concurrent access для многопользовательских сценариев
- Temporal-поля (`valid_from`, `valid_to`) нативно поддерживаются как `DATE` / `TIMESTAMP`
- `pg_dump` для надёжных бэкапов
- Готовность к будущим фичам (полнотекстовый поиск, GIS для мест)

### Отрицательные

- Дополнительный daemon на сервере (PostgreSQL)
- Необходимость настройки `pg_hba.conf`, создания пользователя `timewoven_user`
- Локальная разработка требует запущенный PostgreSQL (или Docker)

### Нейтральные / Требуют внимания

- Файл `family.db` (SQLite) сохранён как архив — откат возможен в течение 30 дней
- Все переменные окружения вынесены в `.env` — переключение движка = замена `DATABASE_URL`
