## 0. Зачем этот документ

Этот документ — единственный источник правды по тому, как мы ведём изменения в TimeWoven. Все остальные документы (TECH_PASSPORT, PROJECT_LOG, PRODUCT_BACKLOG, CHANGELOG, DB_CHANGELOG) описывают **что** в системе. Этот документ описывает **как** мы её безопасно меняем.

Его цель — гарантировать, что после **любого** задания мы можем:

1. вернуть систему ровно в то состояние, в котором она была до задания, одной командой;  
2. никогда не потерять исходники, даже если они ещё не закоммичены;  
3. знать, что текущее состояние сервера, git и документации согласованы.

Если правила этого документа конфликтуют с любым другим — **этот документ имеет приоритет**.

## 1. Базовые принципы

### 1.1. Один блок — один коммит — untracked = 0

Каждое задание (T-XXX) заканчивается **тремя обязательными артефактами**:

1. `git status --porcelain` пуст (нет modified, нет staged, нет untracked, **нет исключений**);  
2. один или несколько связанных коммитов с сообщением вида `T-XXX: <короткое описание>`;  
3. `clean_state_gate.sh` отдаёт `PASS`.

Если задание не закрыто этими тремя артефактами — оно **не закрыто**, независимо от того, работает ли фича.

### 1.2. Untracked-файл — это технический долг, не часть рабочего процесса

Любой `untracked` файл, проживший дольше одного задания, считается дефектом. Допустимы только:

- файлы из `.gitignore` (артефакты сборки, кеши, логи, бэкапы);  
- файлы, явно отложенные через `git add -N` с комментарием в текущем ТЗ.

Никаких «полу-готовых модулей» как untracked: если код должен жить в репо — он должен быть в репо в том же блоке, в котором появился.

### 1.3. `git stash` — не бэкап

Stash может использоваться только как **буфер для switch ветки внутри одного задания** и должен быть закрыт (pop или drop) до конца того же задания. Stash, переживший задание, считается утечкой состояния и должен быть либо закоммичен в feature-ветку, либо явно удалён с записью в PROJECT_LOG.

### 1.4. Документация описывает факт, а не намерение

`TECH_PASSPORT.md` и anchor-файлы описывают **то, что физически есть в репозитории и работает в проде**. Намерения и планы живут только в `PRODUCT_BACKLOG.md`. Если в TECH_PASSPORT упомянут файл — он **обязан** быть в `git ls-files`; несоответствие считается багом документа, а не кода.

### 1.5. Не смешиваем шлифовку и революцию

В одном задании **либо** локальный фикс в существующем scope, **либо** новая архитектурная задача. Если по ходу работы возникает идея, требующая нового UI/новых данных/новых маршрутов — она:

- записывается в `PRODUCT_BACKLOG.md` как новый T-номер;  
- упоминается в `PROJECT_LOG.md` как `deferred during T-XXX`;  
- **не реализуется в текущем задании**.

## 2. Четыре уровня защиты

```text
┌──────────────────────────────────────────────────────┐
│ Уровень 4. Контракт ТЗ → чатВ (DEVIATION-RULE)      │
│ Уровень 3. Один блок — один коммит — untracked = 0  │
│ Уровень 2. Safety Snapshot перед каждым заданием    │
│ Уровень 1. Clean State Gate перед стартом           │
└──────────────────────────────────────────────────────┘
```

Каждый уровень страхует следующий; даже если три провалятся, четвёртый удержит.

---

## 3. Уровень 1 — Clean State Gate

**Правило:** ни одно ТЗ не отправляется чатВ, пока на сервере не выполнен `scripts/ops/clean_state_gate.sh` и не получен ответ `PASS`.

Скрипт выполняет 6 проверок:

| # | Проверка                         | PASS-условие                      |
|---|----------------------------------|-----------------------------------|
| 1 | `git status --porcelain`        | пусто                             |
| 2 | `git stash list`                | пусто (или явно подписан и в TTL)|
| 3 | Текущая ветка                   | `main` или `T-XXX-...`           |
| 4 | `git log origin/$BRANCH..HEAD`  | пусто или коммиты `T-XXX:`       |
| 5 | `systemctl is-active timewoven.service` | `active`                 |
| 6 | `curl -sf https://app.timewoven.ru/health` | `{"status":"ok"}`   |

**Если хоть один пункт — FAIL, новое задание не начинается.** Сначала закрываем долг (коммитим, чистим stash, разруливаем расхождение).

---

## 4. Уровень 2 — Safety Snapshot

**Правило:** перед стартом любого задания на сервере выполняется `scripts/ops/safety_snapshot.sh T-XXX-YYYY-MM-DD-NN`; без этого ТЗ не считается начатым.

Снапшот сохраняет в `/root/projects/TimeWoven_snapshots/<TASK_ID>/`:

- `worktree.tar.gz` — полное рабочее дерево, включая untracked (кроме venv/.git/__pycache__/node_modules);  
- `repo.bundle` — git bundle всех веток и тегов;  
- `db.sql.gz` — pg_dump целевой БД из `.env`;  
- `service.status` — текущий статус сервисов timewoven;  
- `pip-freeze.txt` — состав окружения;  
- `INDEX.log` — строку `<TASK_ID> <git-HEAD-hash> <ISO-timestamp>`.

Главный смысл: в отличие от git/stash, `worktree.tar.gz` сохраняет **untracked** файлы.

### 4.1. Хранение снапшотов

- Rolling 30 дней — `/root/projects/TimeWoven_snapshots/`, авто-удаление по mtime внутри `safety_snapshot.sh`.  
- Защищённые — `/root/projects/TimeWoven_snapshots/protected/`, никогда не удаляются автоматически, туда вручную копируются якорные снапшоты и релизы.

### 4.2. Доступ к снапшотам

- `cat /root/projects/TimeWoven_snapshots/INDEX.log` — список снапшотов;  
- `tar -tzf .../worktree.tar.gz | head` — посмотреть содержимое;  
- `scripts/ops/rollback_to_snapshot.sh <TASK_ID>` — выполнить откат.

## 5. Уровень 3 — «один блок — один коммит — untracked = 0»

**Правило:** задание не считается закрытым, пока не выполнено всё из §1.1.

### 5.1. Формат коммит-сообщений

```text
T-XXX: <короткое императивное описание>

<опциональное тело: что/зачем/что затрагивает>

Refs: PROJECT_LOG.md (T-XXX entry)
```



### 5.2. Чек-лист закрытия задания

```text
T-XXX закрытие:
- [ ] git status --porcelain пуст
- [ ] git stash list пуст
- [ ] py_compile / тесты зелёные
- [ ] systemctl is-active timewoven.service = active
- [ ] /health возвращает 200
- [ ] PROJECT_LOG.md обновлён
- [ ] При схеме БД: DB_CHANGELOG.md обновлён
- [ ] При фичах: TECH_PASSPORT.md / PRODUCT_BACKLOG.md обновлён
- [ ] Создан коммит "T-XXX: ..."
- [ ] safety_snapshot для следующего задания не нужен (предыдущий уже закрыт)
```

Любая незакрытая галочка = задание не закрыто.

### 5.3. Откат не завершает задание

Если при выполнении был сделан rollback — это состояние «blocked», требующее новой итерации ТЗ, а не закрытие задания.

## 6. Уровень 4 — Контракт ТЗ для чатВ

**Правило:** каждое ТЗ для чатВ начинается с фиксированной шапки; без шапки ТЗ невалидно.

### 6.1. Обязательная шапка ТЗ

```text
TASK-ID: T-XXX-YYYY-MM-DD-NN

SCOPE-LOCK:
  Что МОЖНО трогать в этом задании (одно-два предложения, конкретно).

DO-NOT-TOUCH:
  - <файл/модуль/маршрут #1>
  - <файл/модуль/маршрут #2>
  - ...

PRE-CHECK:
  bash scripts/ops/clean_state_gate.sh || exit 1
  bash scripts/ops/safety_snapshot.sh T-XXX-YYYY-MM-DD-NN || exit 1

DEVIATION-RULE:
  Если выполнить точно по ТЗ невозможно (отсутствует файл,
  не сходится контракт, обнаружен побочный риск) —
  STOP. Не подменять scope, не "оптимизировать" молча.
  Вернуть отчёт "BLOCKED: <причина>" и ждать новое ТЗ.

EXIT-CRITERIA:
  - git status --porcelain пуст
  - py_compile зелёный
  - /health 200
  - один коммит "T-XXX: ..."
  - чек-лист закрытия задания заполнен
```



### 6.2. DEVIATION-RULE

При любом расхождении между ТЗ и фактом чатВ обязан остановиться и вернуть BLOCKED-отчёт, не предпринимая автономных действий (включая rollback и hotfix), кроме явно разрешённых emergency‑кейсов.

### 6.3. Запрещённые автономные действия

Без явного разрешения в ТЗ чатВ не имеет права:

- `git reset --hard`;  
- `git stash drop`, `git stash clear`;  
- `git push --force`;  
- массовый `git checkout --` по паттерну;  
- удалять файлы вне явного списка;  
- менять `.env`, `nginx.conf`, systemd unit, `requirements.txt`;  
- трогать миграции БД без разрешения.

### 6.4. Разрешённые действия

- читать файлы, логи, статус;  
- запускать `safety_snapshot.sh` и `clean_state_gate.sh`;  
- останавливать выполнение и возвращать отчёт.

## 7. Жизненный цикл задания

```text
 Автор формулирует идею[2]
        │
        ▼
 Perplexity (главный исследователь)[3]
    собирает ТЗ по шаблону §6.1
        │
        ▼
 Автор отправляет ТЗ чатВ[4]
        │
        ▼
 чатВ выполняет PRE-CHECK[5]
    ├── FAIL → возвращает что чинить, задание не стартует
    └── PASS → продолжает
        │
        ▼
 чатВ создаёт safety_snapshot[6]
        │
        ▼
 чатВ выполняет SCOPE[7]
    ├── всё по плану → continue
    └── deviation → BLOCKED-отчёт, STOP
        │
        ▼
 чатВ выполняет EXIT-CRITERIA[8]
        │
        ▼
 чатВ возвращает чек-лист закрытия[9]
        │
        ▼
 Автор + Perplexity ревью[10]
    ├── OK → задание закрыто
    └── Issues → новая итерация ТЗ
```

## 8. Откат и восстановление

### 8.1. Когда откатываем

- сервис лежит и не поднимается за 5 минут;  
- повреждены данные;  
- задание ушло за пределы scope и его нельзя дочинить;  
- автор явно даёт команду «откат».

### 8.2. Команда отката

```bash
bash scripts/ops/rollback_to_snapshot.sh T-XXX-YYYY-MM-DD-NN
```

Скрипт:

1. создаёт новый safety_snapshot текущего состояния (`<TASK_ID>-PRE-ROLLBACK-<timestamp>`);  
2. останавливает `timewoven.service`;  
3. восстанавливает рабочее дерево из `worktree.tar.gz`;  
4. опционально восстанавливает git‑состояние из `repo.bundle`;  
5. опционально восстанавливает БД из `db.sql.gz` (только по явному флагу);  
6. запускает `timewoven.service`;  
7. проверяет `/health`;  
8. печатает отчёт.

### 8.3. Откат БД

Восстановление БД — только в крайних случаях и только после явного согласия автора, поскольку данные после снапшота будут потеряны.

Раз в неделю (или перед якорным снапшотом) выполняется проверка согласованности TECH_PASSPORT, PROJECT_LOG, PRODUCT_BACKLOG и DB_CHANGELOG с реальным репозиторием и миграциями.

## 10. Роли

### 10.1. Автор (Дмитрий)

- Формулирует идею и приоритет.  
- Согласовывает ТЗ перед отправкой Исполнителю (в проекте участвует можель Cursor ai) Далее - Исполнитель.  
- Принимает или отвергает закрытие задания.  
- Решает, когда делать защищённые снапшоты.

### 10.2. Главный исследователь / Архитектор)

- Превращает идею автора в формальное ТЗ по §6.1.  
- Проверяет, что ТЗ не нарушает SCOPE-LOCK / DO-NOT-TOUCH.  
- Ревьюит чек-листы закрытия.  
- Ведёт post-mortem при инцидентах.  
- При необходимости привлекает исследовательские подзадачи.

### 10.3. Исполнитель (VSCode/Cursor — implementation-агент)

- Исполняет ТЗ строго в SCOPE-LOCK.  
- Соблюдает DEVIATION-RULE.  
- Возвращает чек-лист закрытия.  
- При неопределённости делает STOP + BLOCKED-отчёт.

### 10.4. Формат входного запроса от автора

#### 10.4.1. Запрос к главному исследователю

Используется для архитектурных задач, исследований, разбора инцидентов и дизайна подсистем.

#### 10.4.2. Запрос к implementation-агенту

Автор напрямую не формулирует сложные ТЗ для implementation-агента; все такие ТЗ проходят через главного исследователя.

#### 10.4.3. Запрос к консультативному AI в браузере

Допустим только для быстрых консультаций по концепциям; любые команды и факты о проекте должны проходить верификацию у главного исследователя.

## 11. Дисциплина веток и прод-деплоя

### 11.1. Роли веток

1. Ветка `main`  
   - Единственный источник **production-кода**.  
   - Всё, что развернуто на VPS (timewoven.ru / app.timewoven.ru), соответствует состоянию `main`.  
   - Прямые коммиты в `main` запрещены; изменения попадают через merge из `develop`.

2. Ветка `develop`  
   - Интеграционная ветка.  
   - Сюда попадают изменения из `feature/*` после базовой проверки (lint, тесты, ручной чек).  
   - Может использоваться для отдельного стенда, но не деплоится на production.

3. Ветки `feature/*`  
   - Рабочие ветки для задач: `feature/makefile-phase2`, `feature/landing-waitlist`, `feature/admin-early-access` и т.п.  
   - Любая разработка, правки Makefile, Alembic, UI и т.д. делаются только здесь.  
   - После завершения задачи ветка вливается в `develop` и может удаляться.

### 11.2. Базовый цикл работы с фичей

1. Создание feature-ветки:

```bash
git checkout develop
git pull origin develop
git checkout -b feature/<short-name>
```

2. Работа в feature-ветке:

- Вносятся изменения в код/конфиги/Makefile.  
- Запускаются проверки:

```bash
make lint
make test
make migrate-check   # alembic upgrade head --sql, просмотр SQL
```

- Изменения фиксируются:

```bash
git add <files>
git commit -m "<TaskID>: краткое описание изменения"
```

3. Возврат изменений в `develop`:

```bash
git checkout develop
git pull origin develop
git merge feature/<short-name>      # или PR feature → develop
git push origin develop
```

4. Перенос проверенных изменений в `main`:

```bash
git checkout main
git pull origin main
git merge develop                   # или PR develop → main
git push origin main
```

После этого код в `main` считается готовым к прод-деплою.

### 11.3. Жёсткое правило прод-деплоя

1. Production-сервер всегда работает от ветки `main`.  
2. На сервере запрещено:

- переключать рабочую копию приложения на ветки `develop` или `feature/*`;  
- вносить прямые правки в код, конфиги или шаблоны (любые изменения проходят через git).[web:48]

3. Любой прод-деплой выполняется только после того, как:

- нужные изменения слиты в `develop`;  
- затем слиты из `develop` в `main`;  
- `origin/main` обновлён и прошёл локальный self-check (`make lint`, `make test`, `make migrate-check`).

### 11.4. Алгоритм прод-деплоя (main → VPS)

1. Локальный self-check:

```bash
cd /path/to/TimeWoven

git checkout main
git pull origin main

make lint
make test
make migrate-check
```

2. Подключение к серверу:

```bash
ssh root@193.187.95.221
cd /root/projects/TimeWoven
```

3. Обновление `main` на сервере:

```bash
git checkout main
git status            # должно быть "working tree clean"
git pull --rebase origin main
```

4. Миграции и перезапуск сервиса:

```bash
# при необходимости:
# source venv/bin/activate

make migrate

sudo systemctl restart timewoven.service
sudo systemctl status timewoven.service --no-pager
```

5. Health-check:

```bash
curl -sS http://127.0.0.1:8000/health
```

Ожидается `{"status":"ok"}`; затем ручная проверка в браузере `https://app.timewoven.ru/`.[web:57][web:60]

6. Snapshot:

```bash
make snapshot
```

Фиксация операционного якоря после успешного деплоя.

### 11.5. Быстрый rollback после неудачного деплоя

Если после деплоя:

- сервис не поднимается;  
- /health не возвращает `{"status":"ok"}`;  
- в браузере критичные ошибки, которые нельзя быстро исправить —

то выполняется упрощённый rollback:

1. Найти ID последнего стабильного снапшота (например, по PROJECT_LOG или `INDEX.log`).  
2. На сервере:

```bash
cd /root/projects/TimeWoven
bash scripts/ops/rollback_to_snapshot.sh <STABLE_TASK_ID>
```

3. Дождаться завершения скрипта, проверить:

```bash
systemctl status timewoven.service --no-pager
curl -sS http://127.0.0.1:8000/health
```

4. Зафиксировать инцидент в PROJECT_LOG и завести новое ТЗ на разбор причины (отдельная итерация, не «продолжение» предыдущего задания).
