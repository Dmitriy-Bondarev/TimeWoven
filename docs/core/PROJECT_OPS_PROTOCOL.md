## 0. Зачем этот документ

Этот документ — единственный источник правды по тому, как мы ведём изменения в TimeWoven. Все остальные документы (TECH_PASSPORT, PROJECT_LOG, PRODUCT_BACKLOG, CHANGELOG, DB_CHANGELOG) описывают **что** в системе. Этот документ описывает **как** мы её безопасно меняем.

Его цель — гарантировать, что после **любого** задания мы можем:

1. вернуть систему ровно в то состояние, в котором она была до задания, одной командой;
2. никогда не потерять исходники, даже если они ещё не закоммичены;
3. знать, что текущее состояние сервера, git и документации согласованы.

Если правила этого документа конфликтуют с любым другим — **этот документ имеет приоритет**.

## 1. Базовые принципы

### 1.1. Один блок — один коммит — untracked = 0

Каждое задание (T-XXX) заканчивается **тремя обязательными артефактами**:

1. `git status --porcelain` пуст (нет modified, нет staged, нет untracked, **нет исключений**);
2. Один или несколько связанных коммитов с сообщением вида `T-XXX: <короткое описание>`;
3. `clean_state_gate.sh` отдаёт `PASS`.

Если задание не закрыто этими тремя артефактами — оно **не закрыто**, независимо от того, работает ли фича.

### 1.2. Untracked-файл — это технический долг, не часть рабочего процесса

Любой `untracked` файл, проживший дольше одного задания, считается дефектом. Допустимы только:

- файлы из `.gitignore` (артефакты сборки, кеши, логи, бэкапы);
- файлы, явно отложенные через `git add -N` с комментарием в текущем ТЗ.

Никаких «полу-готовых модулей» как untracked. Если код должен жить в репо — он должен быть в репо в том же блоке, в котором появился.

### 1.3. `git stash` — не бэкап

Stash может использоваться только как **буфер для switch ветки внутри одного задания**, и должен быть закрыт (pop или drop) до конца того же задания. Stash, переживший задание, считается утечкой состояния и должен быть либо закоммичен в feature-ветку, либо явно удалён с записью в PROJECT_LOG.

### 1.4. Документация описывает факт, а не намерение

`TECH_PASSPORT.md` и якоря описывают **то, что физически есть в репозитории и работает в проде**. Намерения и планы живут только в `PRODUCT_BACKLOG.md`. Если в TECH_PASSPORT упомянут файл — он **обязан** быть в `git ls-files`. Несовпадение — баг документа, не кода.

### 1.5. Не смешиваем шлифовку и революцию

(Уже зафиксировано в инструкциях Space, повторяю как обязательное правило.)

В одном задании **либо** локальный фикс в существующем scope, **либо** новая архитектурная задача. Если по ходу работы возникает идея, требующая нового UI/новых данных/новых маршрутов — она:

- записывается в `PRODUCT_BACKLOG.md` как новый T-номер;
- упоминается в `PROJECT_LOG.md` как `deferred during T-XXX`;
- **не реализуется в текущем задании**.

## 2. Четыре уровня защиты

```
┌──────────────────────────────────────────────────────┐
│ Уровень 4. Контракт ТЗ → чатВ (DEVIATION-RULE)       │
│ Уровень 3. Один блок — один коммит — untracked = 0   │
│ Уровень 2. Safety Snapshot перед каждым заданием     │
│ Уровень 1. Clean State Gate перед стартом            │
└──────────────────────────────────────────────────────┘
```

Каждый уровень страхует следующий. Даже если три провалятся, четвёртый удержит.

---

## 3. Уровень 1 — Clean State Gate

**Правило:** ни одно ТЗ не отправляется чатВ, пока на сервере не выполнен `scripts/ops/clean_state_gate.sh` и не получен ответ `PASS`.

Скрипт выполняет 6 проверок:

| # | Проверка | PASS-условие |
|---|----------|--------------|
| 1 | `git status --porcelain` | пусто |
| 2 | `git stash list` | пусто (или явно подписан и в TTL) |
| 3 | Текущая ветка | `main` или `T-XXX-...` |
| 4 | `git log origin/$BRANCH..HEAD` | пусто или ровно подписанные коммиты `T-XXX:` |
| 5 | `systemctl is-active timewoven.service` | `active` |
| 6 | `curl -sf https://app.timewoven.ru/health` | `{"status":"ok"}` |

**Если хоть один пункт — FAIL, новое задание не начинается.** Сначала закрываем долг (коммитим, чистим stash, разруливаем расхождение).

См. `scripts/ops/clean_state_gate.sh`.

---

## 4. Уровень 2 — Safety Snapshot

**Правило:** перед стартом любого задания на сервере выполняется `scripts/ops/safety_snapshot.sh T-XXX-YYYY-MM-DD-NN`. Без этого ТЗ не считается начатым.

Снапшот сохраняет в `/root/projects/TimeWoven_snapshots/<TASK_ID>/`:

- `worktree.tar.gz` — **полное** рабочее дерево, включая untracked (исключая venv/.git/__pycache__/node_modules);
- `repo.bundle` — git bundle всех веток и тегов;
- `db.sql.gz` — pg_dump целевой БД из `.env`;
- `service.status` — текущий статус timewoven сервисов;
- `pip-freeze.txt` — текущий состав окружения;
- `INDEX.log` — строка `<TASK_ID> <git-HEAD-hash> <ISO-timestamp>` (общий журнал снапшотов).

**Главный смысл:** в отличие от git/stash, `worktree.tar.gz` сохраняет **untracked файлы тоже**. Это и есть та защита, которой не было 26.04.

См. `scripts/ops/safety_snapshot.sh`.

### 4.1. Хранение снапшотов

- **Rolling 30 дней** — каталог `/root/projects/TimeWoven_snapshots/`, авто-удаление по mtime скриптом `safety_snapshot.sh` при каждом запуске.
- **Защищённые (protected/)** — `/root/projects/TimeWoven_snapshots/protected/` — никогда не удаляются автоматически. Туда вручную копируются:
  - снапшот перед крупным архитектурным изменением;
  - снапшот раз в неделю как «якорь»;
  - снапшот после стабильного релиза.

### 4.2. Доступ к снапшотам

- `cat /root/projects/TimeWoven_snapshots/INDEX.log` — посмотреть все доступные снапшоты;
- `tar -tzf /root/projects/TimeWoven_snapshots/<TASK_ID>/worktree.tar.gz | head` — посмотреть, что внутри;
- `scripts/ops/rollback_to_snapshot.sh <TASK_ID>` — выполнить откат.

## 5. Уровень 3 — Контракт «один блок — один коммит — untracked = 0»

**Правило:** задание не считается закрытым, пока не выполнено всё из секции 1.1.

### 5.1. Коммит-сообщение — единый формат

```
T-XXX: <короткое императивное описание>

<необязательное тело: что/зачем/что затрагивает>

Refs: PROJECT_LOG.md (T-XXX entry)
```

### 5.2. Чек-лист закрытия задания (обязателен)

В конце каждого задания чатВ обязан вернуть отчёт по фиксированному шаблону:

```
T-XXX закрытие:
- [ ] git status --porcelain пуст
- [ ] git stash list пуст
- [ ] py_compile / тесты зелёные
- [ ] systemctl is-active timewoven.service = active
- [ ] /health возвращает 200
- [ ] PROJECT_LOG.md обновлён
- [ ] При схеме БД: DB_CHANGELOG.md обновлён
- [ ] При фичах: TECH_PASSPORT.md / PRODUCT_BACKLOG.md обновлён
- [ ] Создан коммит "T-XXX: ..."
- [ ] safety_snapshot для следующего задания не нужен (предыдущий уже закрыт)
```

Любая невыполненная галочка = задание не закрыто.

### 5.3. Откат не считается завершением

Если при выполнении что-то сломалось и был сделан rollback — это **не** закрытие задания. Это переход в состояние «blocked», требующий новой итерации ТЗ.

## 6. Уровень 4 — Контракт ТЗ для чатВ

**Правило:** каждое ТЗ для чатВ начинается с фиксированной шапки. Без шапки — ТЗ невалидно и не должно исполняться.

### 6.1. Обязательная шапка ТЗ

```
TASK-ID: T-XXX-YYYY-MM-DD-NN

SCOPE-LOCK:
  Что МОЖНО трогать в этом задании (одно-два предложения, конкретно).

DO-NOT-TOUCH:
  - <файл/модуль/маршрут #1>
  - <файл/модуль/маршрут #2>
  - ...

PRE-CHECK:
  bash scripts/ops/clean_state_gate.sh || exit 1
  bash scripts/ops/safety_snapshot.sh T-XXX-YYYY-MM-DD-NN || exit 1

DEVIATION-RULE:
  Если выполнить точно по ТЗ невозможно (отсутствует файл,
  не сходится контракт, обнаружен побочный риск) —
  STOP. Не подменять scope, не "оптимизировать" молча.
  Вернуть отчёт "BLOCKED: <причина>" и ждать новое ТЗ.

EXIT-CRITERIA:
  - git status --porcelain пуст
  - py_compile зелёный
  - /health 200
  - один коммит "T-XXX: ..."
  - чек-лист закрытия задания заполнен
```

### 6.2. DEVIATION-RULE — главное новое правило

Это **аппаратная** защита от того, что произошло 26.04, когда чатВ молча взял `stash@{2}` вместо `stash@{1}`.

При любом расхождении между текстом ТЗ и фактической ситуацией на сервере чатВ обязан:

1. **Остановиться**.
2. Вернуть отчёт в формате:
   ```
   BLOCKED: <короткая причина>
   FACTS:
     - <что реально на сервере>
   QUESTION:
     - <что мне нужно решить>
   NO ACTION TAKEN.
   ```
3. **Не предпринимать никаких автономных действий**, даже «безопасных» (включая rollback, hotfix, отключение модулей и т.д.) — кроме случаев, когда сервис лежит и явный hotfix согласован в исходном ТЗ.

### 6.3. Запрещённые автономные действия

Без отдельного разрешения в ТЗ чатВ **не имеет права**:

- `git reset --hard`
- `git stash drop`, `git stash clear`
- `git push --force`, `git push --force-with-lease`
- `git checkout -- <массовый паттерн>` (только конкретные пути)
- удалять файлы кроме явно перечисленных в ТЗ
- менять `.env`, `nginx.conf`, `systemd unit`, `requirements.txt` без явного разрешения
- работать с миграциями БД без явного разрешения

### 6.4. Допустимые автономные действия (всегда)

- читать что угодно (`cat`, `git log`, `git show`, `find`, `grep`, `systemctl status`, `journalctl`, `curl`, `psql -c "select ..."` без модификаций);
- запускать `safety_snapshot.sh` и `clean_state_gate.sh`;
- останавливать выполнение и возвращать отчёт.

## 7. Жизненный цикл задания

```
[1] Автор формулирует идею
        │
        ▼
[2] Perplexity (главный исследователь)
    собирает ТЗ по шаблону из секции 6.1
        │
        ▼
[3] Автор отправляет ТЗ чатВ
        │
        ▼
[4] чатВ выполняет PRE-CHECK
    ├── FAIL → возвращает что чинить, задание не стартует
    └── PASS → продолжает
        │
        ▼
[5] чатВ создаёт safety_snapshot
        │
        ▼
[6] чатВ выполняет SCOPE
    ├── всё по плану → continue
    └── deviation → BLOCKED-отчёт, STOP
        │
        ▼
[7] чатВ выполняет EXIT-CRITERIA проверки
        │
        ▼
[8] чатВ возвращает чек-лист закрытия задания
        │
        ▼
[9] Автор + Perplexity ревью
    ├── OK → задание закрыто
    └── Issues → новая итерация ТЗ (НЕ продолжение)
```

## 8. Откат и восстановление

### 8.1. Когда откатываем

- сервис лежит и не поднимается за 5 минут;
- обнаружено повреждение данных;
- задание ушло за пределы scope и невозможно «дочинить» в текущей итерации;
- автор явно скомандовал «откат».

### 8.2. Команда отката

```bash
bash scripts/ops/rollback_to_snapshot.sh T-XXX-YYYY-MM-DD-NN
```

Скрипт:

1. Делает **новый** safety_snapshot текущего состояния (под именем `<TASK_ID>-PRE-ROLLBACK-<timestamp>`) — на случай, если откат сам пойдёт не так;
2. Останавливает `timewoven.service`;
3. Восстанавливает рабочее дерево из `worktree.tar.gz`;
4. Восстанавливает git состояние из `repo.bundle` (опционально, по флагу `--with-git`);
5. Опционально восстанавливает БД из `db.sql.gz` (по флагу `--with-db`, **по умолчанию НЕ восстанавливает**);
6. Запускает `timewoven.service`;
7. Проверяет `/health`;
8. Печатает отчёт.

### 8.3. БД-откат — отдельная процедура

БД восстанавливаем только в крайнем случае — потерянные после снапшота записи восстановить нельзя. Перед `--with-db` обязательно явное согласие автора.

## 9. Документация — что и куда писать

| Документ | Что туда пишем | Когда обновляем |
|----------|----------------|-----------------|
| `PROJECT_LOG.md` | Хронологический журнал заданий | После каждого закрытого T-XXX |
| `PRODUCT_BACKLOG.md` | Будущие задачи, идеи, отложенные изменения | При появлении идеи + при закрытии задания |
| `TECH_PASSPORT.md` | Текущая фактическая архитектура | Только когда фича реально в проде и в git |
| `CHANGELOG.md` | Видимые изменения по версиям | При релизе |
| `DB_CHANGELOG.md` | Изменения схемы БД | При каждой миграции |
| `TimeWoven_Anchor_YYYY-MM-DD.md` | Snapshot состояния системы на дату | Раз в неделю или перед крупным изменением |

### 9.1. Правило согласованности документов

Раз в неделю (или перед якорным снапшотом) выполняется **doc-reconciliation**:

- все упомянутые в TECH_PASSPORT файлы существуют в `git ls-files`;
- все T-номера из PROJECT_LOG имеют статус (Done / Deferred / Cancelled);
- PRODUCT_BACKLOG не содержит T-номеров со статусом Done без даты закрытия;
- DB_CHANGELOG соответствует реальным миграциям в `migrations/`.

Несоответствие — отдельный T-DOC-RECONCILE-... в backlog.

## 10. Роли

### 10.1. Автор (Дмитрий)
- Формулирует идею и приоритет.
- Согласовывает ТЗ перед отправкой чатВ.
- Принимает или отвергает закрытие задания.
- Решает, когда делать защищённые (`protected/`) снапшоты.

### 10.2. Perplexity computer (главный исследователь / архитектор)
- Превращает идею автора в формальное ТЗ по шаблону из 6.1.
- Проверяет, что ТЗ не нарушает SCOPE-LOCK / DO-NOT-TOUCH.
- Ревьюит чек-лист закрытия от чатВ.
- Ведёт post-mortem при инцидентах.
- При необходимости привлекает специализированные исследовательские задачи (research subagents) — не делает большие исследования внутри основной сессии, чтобы не терять контекст.

### 10.3. чатВ (VSCode/Cursor — implementation-агент)
- Исполняет ТЗ строго в SCOPE-LOCK.
- Соблюдает DEVIATION-RULE.
- Возвращает чек-лист закрытия задания.
- При неопределённости — STOP + BLOCKED-отчёт, никогда не «оптимизирует молча».

### 10.4. Формат входного запроса от автора

Когда автор формулирует потребность для главного исследователя или implementation-агента — простыми словами, не техническим языком. Главный исследователь переводит это в формальное ТЗ по шаблону §6.1. Этот раздел — справочник, как удобнее формулировать вход.

#### 10.4.1. Запрос к главному исследователю (Perplexity computer)

Используется для архитектурных задач, исследовательской работы, разбора инцидентов, дизайна новых подсистем.

Нужен дизайн / исследование по задаче <T-номер или короткое имя>.
Контекст: см. <ссылка на anchor / TECH_PASSPORT / PROJECT_LOG>.
Вопросы:

<вопрос 1>

<вопрос 2>
Ожидаемый результат: <дизайн-документ / ТЗ для implementation-агента / решение>


Главный исследователь имеет доступ к файлам Space (документация, anchor-файлы, чаты) и проверяет факты по исходникам, а не по памяти. Если в запросе чего-то критически не хватает — спрашивает, не угадывает.

#### 10.4.2. Запрос к implementation-агенту (VSCode/Cursor) — через главного исследователя

Автор НЕ формулирует ТЗ для implementation-агента напрямую. Этот шаг всегда проходит через главного исследователя, который собирает ТЗ по шаблону §6.1.

Если автор хочет передать что-то напрямую (например, уточнение по уже работающему ТЗ) — формат предельно простой:
Уточнение к T-XXX: <одно-два предложения, что уточняется>

Не более. Любое сложное уточнение оформляется как новое ТЗ через главного исследователя.

#### 10.4.3. Запрос к консультативному AI в браузере

Допустим только для быстрых консультаций «объясни концепцию», «как обычно делают X». Не используется для:
- формулировки ТЗ для implementation-агента,
- проверки фактов о проекте,
- выдачи команд для терминала, которые автор будет копировать на сервер,
- любых действий, изменяющих состояние репозитория или сервера.

Этот тип AI не имеет доступа к актуальным файлам проекта и работает по своему контекстному summary, который может расходиться с реальностью. Все его рекомендации, касающиеся проекта, должны проходить верификацию у главного исследователя.

## 11. Самоконтроль Perplexity

(Введено по итогам моего собственного разбора ошибок 26.04.)

Perplexity computer обязан:

- начинать каждое восстановительное/изменяющее ТЗ с явного `PRE-CHECK` и `safety_snapshot`;
- при любом отклонении чатВ от ТЗ — **остановить работу и переспросить**, а не идти дальше;
- разбивать длинные сессии (> ~60 минут активной работы) на письменно зафиксированные блоки с чек-поинтами, чтобы не терять контекст;
- не путать сущности продукта (например, «карточка персоны» и «экран золотого воспоминания») — при сомнении сверяться с TECH_PASSPORT и Anchor.

После применения этого протокола на сервере должно быть:

## 11. Development Workflow (Git)

### Branch model

- `main` — production branch (всегда deployable)
- `develop` — integration branch (основная ветка разработки)
- `feature/*` — ветки под конкретные задачи

### Rules

1. Прямые коммиты в `main` запрещены.
2. Вся разработка идёт через `feature/*` ветки.
3. Каждая задача = отдельная ветка:
   - `feature/T-XXX-short-name`
4. Слияние происходит:
   - `feature/*` → `develop` (через PR или контролируемый merge)
   - `develop` → `main` (только после проверки)

### Sync with OPS protocol

- Перед началом задачи:
  - `clean_state_gate.sh`
  - `safety_snapshot.sh`
- После завершения:
  - 1 задача = 1 commit
  - `git status` должен быть чистым

### Deviation rule

Любое отклонение от этого workflow = BLOCKED

### GDPR / Data Ownership Rules

- Every family must have an explicit owner (User)
- No data can exist without an owner
- All user data must be exportable
- All user data must be deletable
- Consent must be recorded before storing personal data
