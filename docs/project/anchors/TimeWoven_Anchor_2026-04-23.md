# TimeWoven Anchor — 2026-04-23

**TimeWoven** — цифровая экосистема семейного наследия: воспоминания членов семьи (аудио, текст, фото) привязываются к людям, событиям и местам, формируя живую семейную летопись поверх генеалогического дерева. Это не просто хранилище: каждое воспоминание — субъективная единица смысла, которую можно слушать, читать, переспрашивать.

Этот Anchor-документ — срез состояния проекта на **23.04.2026**: стек, архитектура, данные, последние изменения и то, куда движется работа. Читается за 5–10 минут и заменяет беглый просмотр полдюжины файлов.

---

## 1. Краткое описание проекта

TimeWoven решает задачу сохранения и передачи семейной памяти через несколько каналов:

- **Семейная лента воспоминаний** — голосовые истории, текстовые записи и ответы родственников, опубликованные на `app.timewoven.ru`.
- **Семейный граф** (`/family/tree`) — интерактивное дерево с поддержкой временных слоёв: можно посмотреть, как выглядела семья в конкретный год.
- **Голосовой интерфейс** — запись аудио → транскрипция через Whisper на Mac с M5 Pro → публикация после ревизии администратором.
- **Max-бот** (`app.timewoven.ru/webhooks/maxbot/incoming`) — бот в Max Messenger, через который участники отправляют текстовые сообщения и аудиовоспоминания напрямую в архив.
- **Веб-интерфейс** — Jinja2-шаблоны: публичные страницы семьи, PIN-аутентификация, профили персон, таймлайн, административный интерфейс.
- **AI-анализ** — опциональный провайдер-агностичный анализатор (`ai_analyzer.py`): извлекает людей, даты, локации из текста памяти; переключается через `.env` (`disabled|mock|anthropic|local_stub|llama_local`). Провайдер `llama_local` (T18.A) позволяет Max-боту использовать локальный LLaMA-сервер на Mac M5 Pro через SSH reverse-tunnel (`AI_LLAMA_LOCAL_URL=http://127.0.0.1:19000/analyze`).

**Что реально существует сейчас:** FastAPI-бэкенд, PostgreSQL 14, Nginx/systemd-деплой на VPS, Max-вебхук (M1 и M2 замкнуты), аудио-пайплайн (V1, с участием Mac), семейный граф с temporal-режимом, веб-интерфейс с семейными и административными шаблонами.

## 2. Текущее техническое состояние на 2026-04-23

### База данных
- **СУБД:** PostgreSQL 14+, VPS `193.187.95.221`.
- Схема включает ~15 таблиц: `People`, `People_I18n`, `Unions`, `UnionChildren`, `PersonRelationship`, `RelationshipType`, `Memories`, `MemoryPeople`, `Quotes`, `Events`, `EventParticipants`, `Places`, `AvatarHistory`, **`MaxContactEvents`** (добавлена T16).
- Актуальные миграции: `001`–`005`; последняя — T16 (MaxContactEvents + cleanup тестовых дублей).
- Бэкапы: `pg_dump` в `backups/`, скрипт `scripts/backup_manager.sh`.

### Бэкенд
- **Стек:** Python 3.10.12, FastAPI 0.110+, SQLAlchemy 2.0+, Uvicorn, Jinja2 3.1+.
- **Сервис:** `timewoven.service` под systemd, порт 8000, проксируется Nginx.
- **Живые API:** `/family/*` (дерево, профили, таймлайн, who-am-i), `/admin/*` (CRUD персон, транскрипции, аватары), `/webhooks/maxbot/incoming`, `/api/transcription/*`, `/health`.
- **Безопасность:** PIN-аутентификация для семейных участников, отдельный admin-flow; `security.py` в `app/core/`.

### Max-бот
- Контур M1 (приём → сохранение → ACK) и M2 (AI-анализ текста) **замкнуты**.
- Входящие текстовые сообщения сохраняются в `Memories`; контактные attachment-события — в `MaxContactEvents` (не создают `People` автоматически, T16).
- Маппинг пользователя: по `People.messenger_max_id`; при отсутствии — inbox с `author_id=NULL`.
- Outbound: короткий ACK-ответ через Max API.
- **T18.A:** AI-анализ через `llama_local` — Max-бот может отправлять текст памяти на локальный LLaMA-сервер (Mac M5 Pro) по SSH reverse-tunnel. При недоступности туннеля/сервера Memory сохраняется штатно, анализ пропускается без падения сервиса.

### Активные доработки / недавно завершённые
- **T16 (2026-04-23):** стоп авто-создания `People` из Max-контактов, новый inbox `MaxContactEvents`, cleanup тестовых дублей (`People 35–39 → test_archived`, `Memories 20–24 → archived`).
- **T14 (2026-04-23):** поле `People.record_status` (`active|archived|test_archived`), live UX фильтрует только `active`.
- **T13:** таймлайн показывает только published-записи, скрыты технические blob-payloads.
- **6C.1–6C.3:** прототип temporal snapshot-навигации в семейном графе (keyframe-mode, stable transitions, wheel-навигация).
- **Документация** (T10): `tech-docs/` — центральное хранилище архитектурных решений; `temp/` изолирован в `.gitignore`.

## 3. Ключевые архитектурные решения (ADR overview)

**ADR-001 — Миграция с SQLite на PostgreSQL** *(Accepted, 2026-04-16)*
Проект перешёл на PostgreSQL из-за ограничений concurrent writes и слабой поддержки temporal-типов в SQLite. Итог: продакшен стабилен на pg_dump/pg_restore, открыты JSONB и интервальные запросы, бэкапы автоматизированы.

**ADR-002 — Temporal normalization для PersonRelationship** *(Accepted, 2026-04-17)*
В таблицу `PersonRelationship` добавлены `valid_from`/`valid_to` для моделирования временных отношений (браки с датами, усыновления, изменение ролей). `NULL` = «всегда действующая связь». Итог: запросы вида «кто с кем был в браке на год X» стали тривиальными; Union и PersonRelationship синхронизированы.

**ADR-003 — Аудио-пайплайн: VPS + Mac/Whisper** *(Accepted, 2026-04-18)*
VPS с 1.8 GB RAM не справляется с Whisper; транскрипция вынесена на MacBook Pro M5 Pro (48 GB, Whisper large-v3 / MLX). Пайплайн: загрузка аудио на VPS → rsync на Mac → транскрипция → POST результата → ревизия администратором → публикация. Данные семьи не покидают инфраструктуру.

**ADR-004 — Mac Audio Watcher** *(Accepted, 2026-04-18)*
Lightweight polling-демон (`scripts/watcher.py`) для macOS следит за папкой на Desktop, дедуплицирует файлы по SHA256 и загружает новые аудио на VPS через POST. Решение минималистичное, без heavy-ML-зависимостей на стороне Mac.

**ADR-005 — Union v2 (Proposed)**
Предложен переход к расширенной модели союзов: тип союза (`union_type`), поддержка single-parent союзов, моделирование adoption/guardianship, будущий strict temporal mode. Изменений схемы БД пока нет; решение на стадии предложения.

**ADR-006 — Семейный граф: от year-input к snapshot-навигации (Proposed)**
Переход family graph с прямого ввода года к навигации по temporal snapshots/layers (flat keyframe timeline → stable transitions → опциональный layer-stack). Прототип (6C.3) уже в коде; backend без изменений.

## 4. Модель данных и важные сущности

**`People` + `People_I18n`**
Центральный справочник участников семейного дерева. `People` хранит атрибуты персоны (даты жизни, роль, мессенджер-id, PIN, `record_status`); `People_I18n` содержит имя/отчество/фамилию на каждом языке. Локализация обязательна при создании записи.

**`Unions` + `UnionChildren`**
Моделирует семейные союзы (брак, партнёрство) между парами участников с датами начала/конца. `UnionChildren` связывает союз с детьми. Это основа для построения семейного графа.

**`PersonRelationship` + `RelationshipType`**
Произвольные именованные связи между персонами (крёстные, усыновление, отношения вне пар) с temporal-полями `valid_from`/`valid_to`. `RelationshipType` — нормализованный справочник типов со свойствами симметрии.

**`Memories`**
Воспоминания как центральная единица контента: текст, аудио, транскрипции, эмоциональный тон, приватность. Привязаны к автору (`author_id`), событию и могут иметь parent (ответы/ветки). Статус жизненного цикла: `pending → published → archived`.

**`Events` + `EventParticipants` + `Places`**
Временна́я шкала семейной истории: события с типом, датами и географической привязкой. `EventParticipants` связывает событие с персонами и их ролями.

**`MaxContactEvents`** *(добавлена T16)*
Inbox для contact attachment-событий из Max Messenger. Хранит raw payload, sender/contact ids, имена и статус обработки (`new|matched|merged|archived`). Не создаёт `People` автоматически — разделение ответственности между «кто пишет» и «кого упомянули».

## 6. Актуальные задачи и фокус

**Надёжность Max-интеграции**
Контур M1/M2 работает, но логирование через `logger` не всегда попадает в journalctl (конфигурация Uvicorn). Нужен мониторинг webhook-пути и алерты на падения.

**Contact enrichment pipeline**
`MaxContactEvents` создан как inbox, но matching/merging логика (`matched|merged`) ещё не реализована. Следующий шаг — ручной или полуавтоматический процесс связывания контакта с существующей `People`-записью.

**Аудио-пайплайн: стабилизация и автоматизация**
`scripts/watcher.py` работает в polling-режиме; при нестабильном подключении возможны пропуски. Нужна надёжная очередь и статусный мониторинг для pending/processing аудио.

**UX семейного графа**
Прототип temporal snapshots (6C.3, keyframe-mode) требует доработки UX: плавные переходы, интуитивные жесты, стабильность при быстрой навигации. ADR-006 описывает roadmap.

**Наблюдаемость и диагностика**
В ходе T16-диагностики обнаружено, что стандартный `logger` недостаточно виден в production. Нужна систематизация: структурированное логирование, health-эндпоинты с метриками, возможно Sentry или аналог.

**Модель данных: Union v2**
ADR-005 (Proposed): single-parent союзы, adoption/guardianship, строгий temporal mode. Реализация требует миграции схемы и обновления семейного графа.

**Качество транскрипций**
Текущий пайплайн (Whisper large-v3 на Mac) даёт хороший результат, но шаг ревизии администратором создаёт бутылочное горлышко при росте объёма. Нужна оценка порога quality-score для автопубликации.
