# ADR-007: Multi-Family Data Architecture (Core + Isolated Family Units)

**Status:** Accepted
**Date:** 2026-04-28
**Related:** TECH_PASSPORT.md, PROJECT_OPS_PROTOCOL.md

---

## Context

TimeWoven is initially developed as a single-family system:

* one PostgreSQL database,
* one set of media files (raw / processed / storage),
* data stored inside the project environment.

However, the product vision includes scaling to thousands of families:

* each family has ~200–250 members,
* each family has its own memories, relationships, and media,
* data must remain isolated between families.

Current structure limitations:

* all data is implicitly tied to a single environment,
* no concept of "which family" a dataset belongs to,
* filesystem and database are not aligned with a multi-family model,
* backup and restore cannot be done per family.

---

## Problem

The current architecture does not support multi-family scaling:

* no global registry of families,
* no mapping between a family and its database,
* no mapping between a family and its filesystem storage,
* media and database are not grouped into a single logical unit.

Without a structural change:

* scaling to multiple families will cause data mixing risks,
* backups will become coarse and unsafe,
* migration and deletion of a single family will be difficult,
* system behavior will become unpredictable.

---

## Decision

Introduce a multi-family architecture based on isolated family units and a central core registry.

### 1. Core database

A dedicated database is introduced:

```text
timewoven_core
```

This database is responsible only for system-level management.

---

### 2. Families registry

A `families` table is introduced:

```sql
CREATE TABLE families (
    id UUID PRIMARY KEY,
    slug TEXT UNIQUE NOT NULL,
    name TEXT,
    db_name TEXT NOT NULL,
    data_path TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);
```

---

### 3. Family identifiers

Two identifiers are used:

* UUID — internal, system-level identifier
* slug — external, human-readable identifier (primary operational key)

Example:

```text
slug: bondarev
```

---

### 4. Database per family

Each family is assigned its own PostgreSQL database:

```text
timewoven_{slug}
```

Example:

```text
timewoven_bondarev
```

---

### 5. Filesystem per family

Each family is assigned its own data directory:

```bash
/root/data/timewoven/{slug}/
```

Structure:

```bash
raw/
processed/
storage/
```

---

### 6. Data isolation model

A family is treated as a fully isolated unit:

```text
family = database + filesystem
```

The `families` table acts as the mapping layer between:

* slug
* database
* filesystem

---

## Consequences

### Positive

* strong data isolation between families
* safe per-family backups and restores
* easy migration or deletion of a single family
* clear and scalable system structure
* alignment between database and filesystem

---

### Negative

* increased number of databases (one per family)
* need for migration management across multiple databases
* introduction of a core system layer

---

## Scope

This ADR defines:

* the data architecture for multi-family support
* the separation between core and family data

This ADR does NOT include:

* data migration (handled separately)
* authentication or user management
* API routing or multi-tenant access logic

---

## Implementation Path (planned)

### Phase 1 — Core foundation

* introduce `timewoven_core`
* create `families` table
* define slug-based naming

### Phase 2 — Default family alignment

* map current dataset to a default family (e.g. `default`)
* align filesystem structure

### Phase 3 — Multi-family enablement

* allow creation of new families
* dynamically resolve database and storage paths per request

---

## Notes

This ADR establishes the foundation for scaling TimeWoven
from a single-family system to a multi-family platform.

The key principle is:

> each family is an isolated, self-contained data unit,
> managed through a central registry.
