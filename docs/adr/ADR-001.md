# ADR-001: Миграция с SQLite на PostgreSQL

> **Статус:** Accepted  
> **Дата:** 2026-04-16  
> **Автор:** Дмитрий Бондарев  
> **Теги:** `database` `infrastructure` `migration`

---

## Контекст

TimeWoven изначально использовал SQLite как основную БД. По мере роста проекта проявились ограничения: блокировка файла при записи (concurrent writes), отсутствие типов вроде JSONB/ARRAY/INTERVAL, слабая поддержка миграций и масштабирования. Запуск продакшен‑сервера на VPS (Hostkey) и многопользовательский сценарий потребовали перехода на полноценную СУБД уровня PostgreSQL.

### Движущие силы (Drivers)

- **Concurrency:** SQLite блокирует файл при записи — несколько пользователей одновременно приводят к узкому горлышку.  
- **Масштабируемость:** планируемые фичи (Telegram Bot, визуальное дерево, temporal‑запросы) требуют серьёзной БД.  
- **Типизация данных:** temporal‑поля (`valid_from`, `valid_to`), JSONB для метаданных воспоминаний.  
- **Продакшен‑готовность:** бэкапы (`pg_dump`), репликация, мониторинг, интеграция с экосистемой PostgreSQL.

---

## Рассмотренные варианты

### Вариант A: Остаться на SQLite

| Критерий           | Оценка |
|--------------------|--------|
| Сложность          | Low    |
| Производительность | Ограничена при concurrent writes |
| Поддержка          | Минимальная инфраструктура       |

**Плюсы:**

- Нулевые затраты на миграцию.  
- Простота локальной разработки (один файл).

**Минусы:**

- Потолок масштабирования уже практически достигнут.  
- Нет нормального concurrent‑доступа.  
- Ограниченные типы данных и инструменты.

### Вариант B: PostgreSQL 14+

| Критерий           | Оценка |
|--------------------|--------|
| Сложность          | Medium |
| Производительность | Отличная для текущих и будущих нагрузок |
| Поддержка          | Зрелая экосистема, документация, инструменты |

**Плюсы:**

- Полный concurrent access.  
- Богатые типы данных (JSONB, temporal, arrays).  
- `pg_dump` и стандартные практики бэкапов/репликации.  
- SQLAlchemy 2.0 поддерживает PostgreSQL нативно.

**Минусы:**

- Стоимость миграции данных.  
- Отдельный сервис PostgreSQL на сервере.  
- Требуется настройка прав, пользователей, `pg_hba.conf`.

### Вариант C: MySQL / MariaDB

| Критерий           | Оценка |
|--------------------|--------|
| Сложность          | Medium |
| Производительность | Хорошая |
| Поддержка          | Широкая, но JSONB/temporal менее удобны, чем в PG |

**Плюсы:**

- Популярность, много готовых хостингов.

**Минусы:**

- Менее богатая типизация под текущий use case.  
- JSONB и temporal инструменты слабее PostgreSQL.  
- Нет преимуществ по сравнению с PostgreSQL для TimeWoven.

---

## Решение

**Выбран вариант B: PostgreSQL 14+.**

PostgreSQL закрывает текущие и прогнозируемые потребности TimeWoven. SQLAlchemy 2.0 позволяет сохранить модели неизменными, меняется только `DATABASE_URL`. Temporal‑поля и JSONB критичны для дальнейшей temporal normalization и хранения сложных метаданных воспоминаний.

---

## Последствия (Consequences)

### Положительные

- Полноценный concurrent access.  
- Нативная поддержка дат и temporal‑типов для `valid_from` / `valid_to`.  
- `pg_dump` и стандартные инструменты резервного копирования и восстановления.  
- Готовность к будущим фичам: полнотекстовый поиск, геоданные (места), аналитика.

### Отрицательные

- Поддержка PostgreSQL‑daemon на сервере.  
- Необходимость настройки пользователя `timewoven_user` и security‑политик.  
- Для локальной разработки нужен PostgreSQL (или Docker‑окружение).

### Нейтральные / Требуют внимания

- `family.db` (SQLite) сохранён как архив; откат возможен через период сохранения бэкапов.  
- Все параметры подключения вынесены в `.env`; смена движка — замена `DATABASE_URL`.
