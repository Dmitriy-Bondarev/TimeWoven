# ADR-005: Union v2, Single Parent & Adoption, Temporal Strict Mode

**Status:** Proposed (Future improvement)  
**Date:** 2026-04-22  
**Related:** ADR-002 (Temporal Normalization for PersonRelationship), DB_CHANGELOG 1.0-1.2

## Context

Current union model:

- Table `unions`: `id`, `partner1_id`, `partner2_id`, `start_date`, `end_date`.
- Table `unionchildren`: `union_id`, `child_id`.
- Family graph v1 uses unions as the backbone for:
  - partner edges (person <-> union),
  - child edges (union <-> child).

Temporal model v1 (ADR-002, family graph 5E):

- Unions have `start_date`/`end_date`, temporal logic works for:
  - union active vs inactive by year,
  - partner/child edges active vs inactive by year.
- Frontend in "По году" mode:
  - does not hide people or unions,
  - only changes visual style for inactive unions/edges (lighter color, dashed lines).

New real-world scenarios from live data:

1. Past union that ended in 2006:
   - In "По году" after 2006 union is inactive, but the union node and related people still live on the canvas.
   - For some users this is OK (soft temporal), for others needed "strict cut".
2. Future union (e.g. 2007) when viewing 1970:
   - Union is not yet started, but both partners and (future) child nodes are already visible in the graph.
   - This is acceptable for "soft narrative" mode, but not for strict historical reconstruction.
3. Child with only one known parent (e.g. Марина):
   - Current graph requires a union to connect child to parents.
   - If only one parent is known or if adoption/guardianship is involved, we either:
     - cannot represent the child in the union-based model,
     - or have to create a "fake" union with a dummy/unknown partner.
4. Adoption / guardianship:
   - A child may have:
     - biological parents (one or two),
     - adoptive parents,
     - guardians (temporal, possibly overlapping).
   - Temporal ranges of those relationships must be visible in the timeline and graph.

## Problem

The current union model is too rigid:

- Exactly two partners per union (no single-parent union).
- Child existence in the graph depends on being attached to some union.
- No explicit modeling of adoption / non-biological parenthood.
- Temporal "По году" mode is soft: shows all people and unions with style changes only.
  - This is fine for v1.
  - But we also want a future "strict" mode where:
    - unions and edges that are not yet started or already ended can be hidden,
    - children can be hidden before birth or before adoption, etc.

We must not break existing data or v1 visuals. Any change must be additive and backward-compatible.

## Decision (planned, not implemented yet)

Introduce a conceptual "Union v2" design for future implementation:

1. Union type / role
   - Add a `union_type` (or equivalent) to classify unions:
     - `marriage_legal`
     - `marriage_civil`
     - `cohabitation`
     - `single_parent` (one known parent)
     - `adoption`
     - `guardianship`
   - Keep existing unions as default `marriage_legal` or `marriage_civil` (no data loss).

2. Single-parent union
   - Allow unions where:
     - `partner1_id` is known,
     - `partner2_id` may be `NULL` (or special "unknown" person).
   - Children linked to such unions via `unionchildren` work as today.
   - Temporal logic for the union stays the same (`start_date`/`end_date`).

3. Adoption / guardianship modeling
   - Option A (preferable for simplicity): model adoption as a union with `union_type='adoption'`.
     - One or two adoptive parents as partners.
     - Children linked via `unionchildren`.
     - `start_date` = legal adoption date, `end_date` = adoption termination (if any).
   - Option B: add new relationship types in `person_relationships` (`adoptive_parent`, `adopted_child`) and derive "virtual unions" for the graph. This is more flexible but more complex.

For v2 we lean towards Option A (explicit adoption unions) to keep the graph model unified: all parent-child visual edges go via some union node.

4. Temporal strict mode in the graph
   - Keep two modes:
     - Soft temporal (current v1): show all people/unions, inactive ones are just visually dimmed/dashed.
     - Strict temporal (future v2):
       - hide unions where `year < start_date` or `year > end_date`,
       - hide partner edges and child edges that reference hidden unions,
       - optionally hide children before their birth year.
   - UX: this can be a toggle or advanced mode in the graph UI.

5. Children with one known parent (Марина case)
   - In v2, allow creating:
     - a `single_parent` union with only the mother as partner,
     - attach Марина as a child of this union,
     - later, if father becomes known, allow converting `single_parent` -> `marriage_*` or adding `partner2_id`.
   - In strict temporal mode:
     - before `start_date` of this union, Марина may:
       - be hidden from the graph,
       - or be shown without edges (to be finalized at UX level).

## Consequences

Short term (v1 / current):

- No changes in schema or code.
- Family graph continues to:
  - require unions for children to appear,
  - use soft temporal mode only,
  - support only two-parent unions in data entry.

Medium term (v2):

- Need a separate implementation plan for:
  - adding `union_type` and possibly allowing `partner2_id=NULL`,
  - data migration strategy for existing unions (default types),
  - UI flows to create single-parent and adoption unions,
  - strict temporal mode in frontend (hiding nodes/edges outside active periods).

Risks:

- More complex semantics around parent roles (biological, adoptive, guardian).
- Potential confusion in UI if we mix soft and strict modes without clear explanation.

Mitigations:

- Keep v1 behavior as default.
- Introduce v2 features gradually, with clear labels and explanations in the UI.
- For each new union type, write at least one explicit example (story) in docs and tests.
