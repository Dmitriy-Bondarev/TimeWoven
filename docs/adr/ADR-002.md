# ADR-002: Temporal Normalization для PersonRelationship

> **Статус:** Accepted  
> **Дата:** 2026-04-17  
> **Автор:** Дмитрий Бондарев  
> **Теги:** `database` `domain-model` `temporal`

---

## Контекст

Модель `person_relationships` хранит связи между персонами (bioparent → child, spouse, sibling и др.). В исходном состоянии связи не имели временного измерения: невозможно было выразить, что брак действовал только в интервале (например, 1975–1990), а также отделить «вечные» связи (биологическое родство) от временных (браки). Для корректной временной визуализации семьи и интеграции с `unions` необходимо добавить temporal‑измерение к связям.

### Движущие силы (Drivers)

- **Историческая точность:** семейные связи меняются во времени (браки, разводы, усыновления, смена статусов).  
- **Синхронизация Union ↔ Relationship:** `start_date` / `end_date` в `unions` должны согласовываться с `valid_from` / `valid_to` в spouse‑связях.  
- **Timeline UX:** запросы вида «кто с кем был в браке на дату X» должны быть тривиальными.

---

## Рассмотренные варианты

### Вариант A: Поля valid_from / valid_to в person_relationships

| Критерий           | Оценка |
|--------------------|--------|
| Сложность          | Low    |
| Производительность | Низкая цена (два DATE‑поля) |
| Поддержка          | Общепринятый паттерн temporal‑моделирования |

**Плюсы:**

- Минимальные изменения схемы.  
- Обратная совместимость: `NULL` означает «всегда действующая связь».  
- Простые запросы по интервалу `[valid_from, valid_to)`.

**Минусы:**

- Требуется аккуратный backfill существующих данных.

### Вариант B: Отдельная таблица relationship_periods

| Критерий           | Оценка |
|--------------------|--------|
| Сложность          | High   |
| Производительность | JOIN при каждом запросе |
| Поддержка          | Сложнее для понимания и сопровождения |

**Плюсы:**

- Возможность хранить несколько периодов для одной связи.

**Минусы:**

- Избыточная нормализация для текущего масштаба.  
- Усложнение запросов, риск «over‑engineering» на ранней стадии.

---

## Решение

**Выбран вариант A: добавление полей `valid_from` и `valid_to` в `person_relationships` и переход к valid‑time модели.**

Поле `valid_from` описывает начало действия связи, `valid_to` — конец (интервал `[from, to)`).  
Открытый интервал реализуется через `valid_to = '9999-12-31'`, `NULL` в качестве конца периода не используется (для упрощения запросов и интервалов).

---

## Последствия (Consequences)

### Положительные

- Можно выразить временные границы браков, партнёрств и других связей.  
- Запросы «состояние семьи на дату X» сводятся к единому предикату `valid_from <= X < valid_to`.  
- Подготовка к визуальному дереву с временной шкалой и к дальнейшей интеграции `unions`.

### Отрицательные

- Требуется аккуратный backfill `valid_from` / `valid_to` на реальных данных.  
- Необходимо синхронно обновлять temporal‑поля при изменении данных (`death_date`, `unions.*`).

### Нейтральные / Требуют внимания

- Семантика разных типов связей (родство vs брак) должна быть чётко задокументирована.  
- При дальнейшем росте может потребоваться хранить несколько интервалов для одной пары персон.
