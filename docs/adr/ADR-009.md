# ADR-009: User Ownership and GDPR Compliance Foundation

**Status:** Proposed  
**Date:** 2026-04-29  
**Related:** ADR-002 (Temporal Model), ADR-008 (Deploy Webhook HMAC)

---

## Context

TimeWoven is evolving into a worldwide product that processes personal and family data.

Current architecture:

- data is stored per-family (1 DB = 1 family),
- no explicit user entity exists,
- no formal ownership model is defined,
- no GDPR-compliant interfaces (consent, export, deletion).

This creates a legal and architectural gap for international usage.

---

## Problem

The system lacks a defined "Data Controller" and user-level ownership:

- no explicit user identity controlling family data,
- no consent tracking,
- no mechanism for data export,
- no mechanism for account deletion.

This prevents compliance with GDPR and similar regulations.

---

## Decision

Introduce a minimal GDPR-compliant user and ownership model.

---

### 1. User Entity

A new entity `User` is introduced:

- id
- email
- created_at
- consent_given_at

---

### 2. Ownership Model

Each family must have a single owner:

User → owns → Family

All data within the family database is considered owned by that user.

---

### 3. Consent Model

The system must record:

- timestamp of consent acceptance
- version of privacy policy (future extension)

---

### 4. Data Access Rights

Users must be able to:

- export all their data
- request deletion of their account and associated data

---

### 5. Deletion Rule

Deleting a user must result in:

- deletion of the associated family database
- deletion of all related media

---

## Consequences

### Positive

- enables GDPR compliance
- defines clear data ownership
- aligns architecture with global standards

---

### Negative

- introduces additional layer (User)
- requires updates in authentication and routing
- adds complexity to data lifecycle management

---

## Scope

This ADR defines:

- user entity
- ownership model
- GDPR minimal compliance layer

This ADR does NOT include:

- UI implementation details
- legal text of privacy policy
- full consent management system

---

## Implementation Path

### Phase 1 — Model Introduction

- introduce User entity
- define User ↔ Family relationship

---

### Phase 2 — Consent Tracking

- record consent timestamp
- introduce minimal consent storage

---

### Phase 3 — Data Control

- implement export endpoint
- implement deletion endpoint

---

## Notes

This ADR establishes the minimal legal foundation required for worldwide usage.

Future extensions may include:

- multi-user families
- role-based access
- advanced consent management
