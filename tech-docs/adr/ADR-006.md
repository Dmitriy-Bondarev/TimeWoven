# ADR-006: Temporal Layers and Snapshot Navigation for Family Graph

**Status:** Proposed (Future improvement)  
**Date:** 2026-04-22  
**Related:** ADR-002 (Temporal Normalization for PersonRelationship), ADR-005 (Union v2, Single Parent & Adoption, Temporal Strict Mode), Family Graph v1

## Context

Current family graph behavior:

- Family graph v1 already renders people, unions and edges in D3.js.
- Backend supports year-based temporal evaluation for unions and edges.
- Frontend has a "Сейчас" / "По году" mode and can request a graph snapshot for a specific year.

However, the current UX is still input-driven:

- The user chooses a year through UI controls.
- The graph is effectively treated as one structure with a year filter.
- The interaction does not yet communicate the idea of historical layers or temporal storytelling.

Product observation:

- A more natural mental model is not "pick a year", but "move through layers of time".
- Each layer is a snapshot of the family system at a particular year or event boundary.
- User should be able to scroll / swipe through these layers and observe how family structure changes:
  - unions appear and end,
  - children appear,
  - active vs inactive ties shift,
  - later, adoptions / guardianship can appear as well.

This matches the core product identity of TimeWoven better than a plain numeric filter.

## Problem

The current year selector is technically functional but conceptually weak:

- it behaves like a filter control,
- it does not create a sense of temporal depth,
- it does not help the user understand historical transitions,
- it does not yet support natural navigation by wheel / touchpad / swipe.

Additionally, naive year switching can cause graph instability:

- if layout is recalculated too aggressively, nodes "jump",
- users lose the sense of continuity between neighboring snapshots.

## Decision (planned, not implemented yet)

Introduce a future "Temporal Layers / Snapshot Timeline" model for the family graph.

### 1. Snapshot-first navigation

The graph should conceptually expose a sequence of temporal snapshots rather than a plain year input.

A snapshot is:

- a graph state for a selected year or event boundary,
- including visible nodes, unions, and edges plus their temporal status.

### 2. Natural temporal navigation

Primary interaction should become:

- mouse wheel / touchpad scroll,
- swipe on touch devices,
- optional keyboard arrows,

instead of relying only on manual year entry.

The existing year input may remain as fallback / precision control.

### 3. Keyframe-based timeline

The preferred timeline should be based not only on raw calendar years, but on meaningful temporal milestones where graph structure changes, such as:

- person birth,
- union start,
- union end,
- death,
- adoption / guardianship start or end (future).

This avoids long runs of empty years and creates a more meaningful user journey.

### 4. Stable graph continuity

Adjacent snapshots should preserve layout continuity as much as possible:

- same people should remain visually near their prior positions,
- only changed nodes/edges should animate in/out or change state,
- transitions must feel like the same family seen across time, not like unrelated redraws.

### 5. Two levels of temporal UX

Future graph UX may include:

- Soft temporal mode:
  all known people remain visible; temporal changes are shown through styling.
- Strict snapshot mode:
  only entities valid for the selected snapshot are shown.

ADR-005 remains the place for future data-model extensions around single-parent unions, adoption and guardianship. ADR-006 focuses on navigation and visualization.

## Consequences

Short term:

- no immediate backend schema change is required;
- current year mode remains valid as baseline functionality;
- snapshot timeline work should begin as UX / frontend exploration, not as DB migration.

Medium term:

- frontend will need a dedicated timeline interaction layer;
- D3 graph update logic must support stable transitions across snapshots;
- backend may later expose "keyframe years" for the current graph scope to optimize navigation.

Risks:

- over-designing a visually impressive but confusing timeline;
- graph instability between snapshots;
- high implementation complexity if 3D depth effects are attempted too early.

Mitigations:

- implement in stages;
- start with flat keyframe timeline + wheel/swipe navigation;
- only later explore perspective/layer-stack visuals.

## Implementation path (planned)

### Phase 1 — Flat snapshot timeline

- Add keyframe timeline UI.
- Support wheel / touchpad / swipe navigation.
- Keep current graph rendering mostly intact.

### Phase 2 — Stable temporal transitions

- Reduce layout jumping between adjacent snapshots.
- Animate changes in unions / child links / node emphasis.

### Phase 3 — Layer-stack visualization

- Explore visual depth inspired by stacked temporal layers.
- Keep it as optional enhancement, not as a prerequisite for usability.

## Notes

This ADR intentionally does not mandate a 3D implementation.
The core decision is conceptual:
TimeWoven family graph should evolve from "graph with year input"
to "navigable temporal snapshot system".
